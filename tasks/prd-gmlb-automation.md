# PRD — GMLB-automation: Имиджевый и лидогенерационный сайт

**Версия:** 1.0
**Дата:** 2026-02-28
**Статус:** Готов к разработке
**Автор:** GMLB

---

## 1. Введение / Обзор

### Проблема

Продукты автоматизации бизнеса (Telegram-боты, парсеры, RAG-решения, чат-боты для маркетплейсов и т.д.) сложно продавать без наглядной онлайн-точки входа. Потенциальные клиенты из сегмента малого бизнеса и e-commerce не понимают ценности и механики работы продуктов без детальной презентации. Нет инструмента для автоматического сбора и обработки входящих заявок.

### Решение

Создать имиджевый, SEO-оптимизированный продуктовый сайт на базе Next.js + Payload CMS, который:
- Профессионально и технологично презентует каждый продукт
- Автоматически собирает лиды и отправляет их в Telegram
- Поддерживает онлайн-оплату через ЮKassa
- Позволяет масштабировать каталог продуктов без участия разработчика

### Целевой рынок

Малый бизнес, e-commerce, реселлеры электроники, владельцы Telegram-магазинов, маркетплейс-продавцы — Россия и СНГ. Язык сайта: **русский**.

---

## 2. Цели

1. **Лидогенерация:** Получать минимум 10+ квалифицированных заявок в месяц через формы и кнопки на сайте.
2. **Имидж:** Сформировать образ технологичной и профессиональной компании; сайт должен восприниматься наравне с зарубежными SaaS-продуктами.
3. **SEO:** Попасть в ТОП-10 Яндекс и Google по целевым ключевым запросам для каждого продукта в течение 6 месяцев после запуска.
4. **Масштабируемость:** Обеспечить возможность добавления новых продуктов через админку без правок в коде — старт с 7–10 продуктов, цель — 50+ в течение года.
5. **Автономность:** После запуска весь контент управляется через CMS без участия разработчика.
6. **Монетизация:** Принимать предоплату онлайн через ЮKassa с мгновенным уведомлением в Telegram.

## 4. Функциональные требования

### 4.1 Главная страница

**FR-01.** Система должна отображать Hero-секцию с крупным заголовком, подзаголовком, CTA-кнопкой («Получить консультацию» / «Смотреть продукты»), анимированным фоном и иллюстрацией.

**FR-02.** Система должна отображать секцию «О компании» с описанием задач, которые решает GMLB.

**FR-03.** Система должна отображать каталог продуктов в виде карточек (glassmorphism-стиль), подтягиваемых из CMS. Каждая карточка содержит: обложку, название, краткое описание, тег категории и кнопку «Подробнее».

**FR-04.** Система должна отображать секцию «Кейсы» с результатами клиентов.

**FR-05.** Система должна отображать секцию «Преимущества» (почему GMLB).

**FR-06.** Система должна отображать секцию «Отзывы» клиентов.

**FR-07.** Система должна отображать блок записи на консультацию с формой лидогенерации.

**FR-08.** Система должна отображать футер с навигацией, контактами и формой.

---

### 4.2 Страница продукта `/products/[slug]`

**FR-09.** Система должна динамически генерировать страницу для каждого продукта из CMS по уникальному slug.

**FR-10.** Страница продукта должна содержать навигацию по вкладкам:
- **Обзор** — краткое описание, обложка, ключевые возможности
- **Боли** — проблемы, которые решает продукт (rich text из CMS)
- **Решение** — как продукт решает эти проблемы (rich text из CMS)
- **Как работает** — пошаговое описание или схема (rich text из CMS)
- **Кейсы** — связанные кейсы из коллекции Cases
- **Тарифы** — ценовые пакеты с диапазоном priceFrom–priceTo
- **FAQ** — часто задаваемые вопросы (из CMS)

**FR-11.** На каждой странице продукта должна быть форма лидогенерации (имя, телефон, Telegram, сообщение) с кнопкой «Оставить заявку».

**FR-12.** На каждой странице продукта должна быть кнопка «Написать в Telegram» (прямая ссылка на Telegram-аккаунт менеджера).

**FR-13.** На каждой странице продукта должна быть кнопка «Оплатить предоплату» с переходом к форме оплаты через ЮKassa.

**FR-14.** Страница продукта должна поддерживать отображение YouTube-видео (если указано в CMS).

**FR-15.** Страница продукта должна отображать галерею изображений (если загружены в CMS).

---

### 4.3 Лидогенерация

**FR-16.** Система должна принимать заявки через форму с полями: Имя (обязательно), Телефон (обязательно), Telegram (необязательно), Сообщение (необязательно), а также скрытое поле — идентификатор продукта (если заявка со страницы продукта).

**FR-17.** При успешной отправке формы система должна:
1. Сохранить лид в коллекции `Leads` в базе данных
2. Отправить уведомление в Telegram-бота в формате:
```
🔔 Новый лид
Продукт: [название или «Главная»]
Имя: [имя]
Телефон: [телефон]
Telegram: [telegram или «не указан»]
Сообщение: [текст или «—»]
```
3. Показать пользователю сообщение об успешной отправке.

**FR-18.** Система должна валидировать форму на стороне клиента (обязательные поля, формат телефона) и на стороне сервера.

**FR-19.** Система должна защищать форму от спама (honeypot-поле или rate limiting на API-роуте).

---

### 4.4 Онлайн-оплата (ЮKassa)

**FR-20.** Система должна создавать платёж через API ЮKassa при нажатии кнопки «Оплатить предоплату» с фиксированной минимальной суммой (задаётся в конфиге).

**FR-21.** Система должна обрабатывать webhook от ЮKassa при успешной оплате:
1. Записать информацию об оплате в базу данных (сумма, статус, дата, ID транзакции)
2. Обновить статус лида в коллекции `Leads` (если лид связан с оплатой)
3. Отправить уведомление в Telegram:
```
✅ Новая оплата
Продукт: [название]
Сумма: [сумма] ₽
Клиент: [имя]
ID транзакции: [id]
```

**FR-22.** Система должна поддерживать страницу успешной оплаты (`/payment/success`) и страницу ошибки (`/payment/cancel`).

### 4.6 SEO

**FR-27.** Каждая страница должна иметь уникальные `<title>` и `<meta name="description">` — берутся из CMS или генерируются автоматически по шаблону.

**FR-28.** Система должна генерировать Open Graph теги (`og:title`, `og:description`, `og:image`) для всех страниц.

**FR-29.** Система должна генерировать `sitemap.xml` со всеми опубликованными страницами продуктов и статей блога.

**FR-30.** Система должна содержать `robots.txt` с корректными правилами для поисковых роботов.

**FR-31.** Все страницы должны иметь корректную структуру заголовков H1–H3.

**FR-32.** Система должна использовать Schema.org разметку (JSON-LD):
- `Organization` — для компании (главная страница)
- `Product` — для страниц продуктов
- `FAQPage` — для FAQ-блока на странице продукта
- `Article` — для статей блога

**FR-33.** Все изображения должны иметь атрибуты `alt`, загружаться с `lazy loading` и быть оптимизированы через Next.js Image.

**FR-34.** URL-структура должна быть человекочитаемой (ЧПУ):
- Главная: `/`
- Продукт: `/products/[slug]`
- Блог: `/blog`, `/blog/[slug]`
- Оплата: `/payment/success`, `/payment/cancel`

---

### 4.7 Производительность

**FR-35.** Сайт должен использовать подход Mobile-first при вёрстке.

**FR-36.** Страницы продуктов должны генерироваться через SSG (Static Site Generation) или ISR (Incremental Static Regeneration) для максимальной скорости загрузки.

**FR-37.** Core Web Vitals должны соответствовать порогу «Good» в Google PageSpeed Insights (LCP < 2.5s, CLS < 0.1, FID < 100ms).

## 5. Не-цели (Вне рамок проекта — v1.0)

Следующее **не входит** в текущую версию и не должно реализовываться:

- **Личный кабинет клиента** — клиенты не регистрируются на сайте.
- **CRM-интеграция** — лиды хранятся в БД и отправляются в Telegram; интеграция с AmoCRM/Bitrix24 — в будущих версиях.
- **Мультиязычность** — сайт только на русском языке; i18n закладывается в архитектуру, но не реализуется.
- **Дополнительные платёжные системы** — только ЮKassa; Stripe, Robokassa и другие — в будущих версиях.
- **Маркетплейс / магазин с корзиной** — сайт лидогенерационный, не интернет-магазин.
- **Аналитическая дашборд-система** — базовая аналитика через Яндекс.Метрику (внешний скрипт), собственная аналитическая панель — в будущих версиях.
- **A/B тестирование** — не реализуется.
- **Автоматическое SEO-управление через AI** — контент создаётся вручную через CMS.

## 6. Дизайнерские решения

### Общий стиль

| Параметр | Значение |
|---------|---------|
| Тема | Тёмная (dark mode) |
| Референс | Vercel.com |
| Стиль карточек | Glassmorphism (полупрозрачный фон, размытие) |
| Акцентный цвет | Синий / голубой (#3B82F6, #60A5FA) |
| Вторичный эффект | Лёгкое неоновое свечение (box-shadow с синим цветом) |
| Шрифт | Inter или Geist (системный стек) |
| Иллюстрации | Плоские технологичные (не обязательно 3D) |

### UI-элементы

- **Кнопки:** Градиентный фон (синий → фиолетовый / синий → голубой), border-radius: 8–12px, hover-эффект со свечением.
- **Карточки продуктов:** `backdrop-filter: blur`, тонкая граница `border: 1px solid rgba(255,255,255,0.1)`, hover-подъём с тенью.
- **Анимации:** Framer Motion — анимация появления блоков (fadeInUp при скролле), плавные переходы вкладок, hover-эффекты на карточках и кнопках.
- **Навигация:** Sticky header, мобильное меню (бургер).
- **Форма:** Тёмные инпуты с акцентным фокусом, inline-валидация, состояние загрузки на кнопке.

### Адаптивность

- Breakpoints: mobile (< 768px), tablet (768–1024px), desktop (> 1024px).
- Сетка продуктов: 1 колонка (mobile) → 2 колонки (tablet) → 3 колонки (desktop).
- Вкладки продукта на мобильном: горизонтальный скролл.

## 7. Технические соображения

### Стек технологий

| Компонент | Технология |
|---------|----------|
| Frontend | Next.js 14+ (App Router), TypeScript |
| Стилизация | Tailwind CSS |
| Анимации | Framer Motion |
| CMS / Backend | Payload CMS 2.x |
| База данных | PostgreSQL (рекомендуется для надёжности) или MongoDB |
| Хостинг | VPS Beget (Node.js процесс) |
| Платежи | ЮKassa REST API |
| Уведомления | Telegram Bot API (через `node-telegram-bot-api` или прямые fetch-запросы) |
| Email | Опционально — Resend или SMTP для дублирования уведомлений |
| Изображения | Next.js Image + Payload Media |

### Ключевые технические решения

**TR-01. Payload CMS как единый бэкенд.**
Payload запускается в том же Next.js проекте (monorepo). API-роуты Payload используются для сохранения лидов и управления коллекциями. Это упрощает деплой на VPS — один Node.js процесс.

**TR-02. Страницы продуктов — ISR.**
Использовать `generateStaticParams` + `revalidate: 60` для страниц продуктов. При добавлении нового продукта через CMS страница автоматически пересоздаётся без полного ребилда.

**TR-03. API-роуты для форм и вебхуков.**
- `POST /api/leads` — приём заявок, валидация, сохранение в Payload, отправка в Telegram.
- `POST /api/payment/create` — создание платежа в ЮKassa.
- `POST /api/payment/webhook` — обработка вебхука ЮKassa (проверка подписи обязательна!).

**TR-04. Защита API.**
- Webhook от ЮKassa верифицируется через проверку HMAC-подписи заголовка `X-Request-Id`.
- Админка Payload защищена email+password аутентификацией.
- API-роуты форм защищены honeypot-полем и rate limiting (например, `upstash/ratelimit` или middleware).

**TR-05. Переменные окружения.**
Все чувствительные данные хранятся в `.env.local`:
```
PAYLOAD_SECRET=
DATABASE_URI=
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
YOKASSA_SHOP_ID=
YOKASSA_SECRET_KEY=
NEXT_PUBLIC_SITE_URL=
```

**TR-06. Процесс менеджер на VPS.**
Использовать `pm2` для управления Node.js процессом на Beget VPS. Настроить автозапуск при перезагрузке сервера.
